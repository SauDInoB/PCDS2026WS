Why Python?

What is Programming?

Writing instructions that a computer can execute, step by step.

  • A program is a recipe: precise, unambiguous, repeatable
  • Python is a language ‚Äî a way to write those instructions

You don‚Äôt need to be a mathematician. You need to be able to think logically.

Analogy: a cooking recipe. If you tell a chef ‚Äúadd some salt‚Äù, they use judgment. A computer cannot ‚Äî you must be exact. That precision is what programming teaches.

Why Python?

Easy to read

# Find patients over 65
elderly = [p for p in patients
           if p.age > 65]

Almost reads like English.

Huge ecosystem

  • pandas ‚Äî data tables
  • numpy ‚Äî numerical computing
  • scipy ‚Äî statistics
  • matplotlib / seaborn ‚Äî plots
  • scikit-learn ‚Äî machine learning
  • streamlit ‚Äî web apps

Show the code block ‚Äî even a non-programmer can guess what it does. That readability lowers the barrier to entry enormously.

Installing Python

The Terminal ‚Äî Your New Best Friend

The terminal (Command Prompt / PowerShell on Windows, Terminal on macOS) lets you talk to your computer with text commands.

Open it:

  • Windows ‚Üí search ‚ÄúCommand Prompt‚Äù or ‚ÄúPowerShell‚Äù

Essential commands:

What is a Virtual Environment?

A virtual environment isolates packages per project ‚Äî no conflicts between projects.

# Activate it (macOS/Linux)
source .venv/bin/activate

# Activate it (Windows)
.venv\Scripts\activate

# Install packages inside it
pip install pandas numpy

Git & GitHub

Why Version Control?

‚ÄúWhich file is the final version?‚Äù

analysis_final.py
analysis_final_v2.py
analysis_final_FINAL.py
analysis_final_FINAL_use_this_one.py  ‚Üê üòÖ

Version control solves this:

  • Every change is tracked with a message
  • You can go back to any point in history
  • Multiple people can work on the same project
  • Nothing is ever truly lost

Every researcher recognises the ‚Äúfinal_FINAL‚Äù problem. This is the emotional hook ‚Äî version control is the professional solution.

GitHub.com

GitHub is a cloud platform for hosting Git repositories.

  • Public repos ‚Äî open source, visible to everyone
  • Private repos ‚Äî only you (and collaborators) can see them
  • Great for sharing analysis code with collaborators or reviewers
  • Journals increasingly require code availability ‚Üí GitHub is the standard

‚Üí Create a free account at github.com

Mention that GitHub is owned by Microsoft. For health data: never put patient data in a GitHub repo ‚Äî only code. Data stays local or in a secure server.

GitHub Desktop

GitHub Desktop makes Git visual ‚Äî no command line needed.

Install: desktop.github.com

Basic workflow:

  1. Create a new repository (or Clone an existing one)
  2. Edit files in your project
  3. GitHub Desktop shows what changed
  4. Write a short commit message ‚Üí Commit to main
  5. Push to sync with GitHub.com

Live demo: create a new repo in GitHub Desktop, create a Python file in Positron, go back to GitHub Desktop, show the diff, write a commit message, commit, push. Open github.com to show it live online.

Best Practices

Why Best Practices Matter

Code is read many more times than it is written.

‚ÄúAlways write code as if the person who will maintain it is a violent psychopath who knows where you live.‚Äù ‚Äî Martin Golding

More practically: future you will thank present you.

Lighten the mood with the quote, then bring it back to reality. In health research, code may need to be audited, reproduced, or handed over to colleagues.

PEP 8 ‚Äî Python‚Äôs Style Guide

PEP 8 is the official Python style guide. Key rules:

# ‚úÖ Good
patient_age = 65
MAX_RETRIES = 3

def calculate_bmi(weight_kg, height_m):
    return weight_kg / height_m ** 2

# ‚ùå Avoid
PatientAge = 65        # CamelCase for variables
x = 65                 # cryptic names
def calc(w,h): return w/h**2  # no spaces, unclear

The point isn‚Äôt memorising PEP 8 ‚Äî Positron and VS Code flag violations automatically. The point is awareness.

Naming Conventions

What Convention Example
Variables & functions snake_case patient_id, load_data()
Constants UPPER_SNAKE_CASE MAX_AGE = 120
Classes PascalCase PatientRecord
Private (internal) _leading_underscore _raw_data

Rule of thumb: names should explain what something is, not how it‚Äôs stored.

# ‚ùå df2      ‚Üí what is this?
# ‚úÖ filtered_patients  ‚Üí clear!

Comments & Docstrings

# Single-line comment: explain the WHY, not the WHAT

# ‚ùå Bad: obvious comment
x = x + 1  # add 1 to x

# ‚úÖ Good: explains intent
age += 1  # adjust for 0-indexed age groups in the dataset

def normalise_score(score, max_score):
    """
    Normalise a raw score to a 0‚Äì1 scale.

    Parameters
    ----------
    score : float
        Raw score value.
    max_score : float
        Maximum possible score.

    Returns
    -------
    float
        Normalised score between 0 and 1.
    """
    return score / max_score

Docstrings are picked up by Positron‚Äôs hover documentation and by tools like Sphinx for auto-generating docs. Encourage writing them from day one.
